## 一、核心概念

```
每季度 = 5 分钟 = 3 点行动力 (AP)

你有一家公司，每个季度产生利润。
利润可以分掉（进你口袋），也可以留下（扩大公司）。
每季度你有 3 点 AP 用来做经营决策。
```

## 三、AP 成本体系

| 消耗 | 类别 | 包含动作 |
|------|------|---------|
| **1 AP** | 经营调整 | 扩产、招人、研发、降本、促销 |
| **0 AP** | 资本操作 | 分红、回购、营销 (只消耗现金不消耗 AP) |
| **2-3 AP** | 战略级 | 并购(2AP)、转型(3AP) |
| **纯现金** | 特殊 | 增发 (不耗 AP，但增加股本稀释) |

**关键设计：** 资本类操作不耗 AP，意味着你可以同时做「扩产 + 分红 + 回购」花 1AP 实现三个效果。但现金是另一个约束——你有钱同时做吗？

## 四、全部行动明细

### 4.1 经营类 (1 AP)

| 行动 | 效果 | 约束 | 行业差异 |
|------|------|------|---------|
| 🏭 扩产 | 新建一条天花板单元（产线/矿井/服务器等） | 现金足够建造成本 | 各行业建造成本和建造周期见 §7.1 |
| 👥 招人 | 员工 +2~5人，成本 ¥3k/人 | 现金足够 | 科技 ¥5k/人但附带研发加成 |
| 🔬 研发 | 利润率 +3%，研发等级 +0.03 | 每季度限 1 次 | 科技/医疗效果 +5%（利润率+5%） |
| 🔧 降本 | 利润率 +4%，员工 -10% | 至少保留 3 人 | 制造额外 +1% |
| 📦 促销 | 立即清 50% 库存，利润率 -5% | 有库存 | 消费行业清 80%，不扣利润率 |

### 4.2 资本类 (0 AP，纯现金)

> **⚠️ 已实现与设计差异**: 当前系统无 AP 分类，营销已实现为**1 次操作**（占每季 3 次限额）、金额可调、当季即刻生效。以下为原设计，分红/回购尚未实现。

| 行动 | 效果 | 成本 | 冷却 |
|------|------|------|------|
| 💰 分红 | 股价 +6%，个人获现金 | ¥10,000~¥20,000（取决于利润） | 无 |

## 五、董事会（抽象满意度）

### 5.1 唯一指标

```
董事会满意度：0-100 的一个数字

每年 (4 季度) 考核一次：
  - 前 2 年 (Q1-Q8) 为保护期：KPI 不达标不扣满意度，仅提示
  - 保护期后：
      ＞ 满意 (≥70)：继续授权，满意度 +8，AP 上限 +1
      ＞ 警告 (40-69)：满意度 -5，KPI 不变
      ＞ 不满 (<40)：满意度 -12，下年仍不达标 → 罢免
  - 满意度 < 20 且不在保护期 → 直接罢免

AP 上限从 3 起，每年考核通过 +1 (最高 5)
```

### 5.2 年度 KPI

注册公司时随机抽取一个 KPI，每年考核后可能更换：

| KPI 类型 | 达标线 | 超额奖励 |
|---------|--------|---------|
| 股价涨幅 | ≥ 15%/年 | 每超 5%，满意度额外 +5 |
| 营收增长 | ≥ 20%/年 | 每超 5%，满意度额外 +3 |
| 利润率 | ≥ 25% | 每超 5%，满意度额外 +4 |
| 分红总额 | ≥ 利润 × 30% | 每超 10%，满意度额外 +3 |
| 员工规模 | ≥ 50 人 | 每超 10 人，满意度额外 +2 |

**设计意图：** KPI 是随机的，你在注册时不知道会拿到什么。拿到「分红」KPI 的科技公司 CEO 和拿到「利润率」KPI 的消费公司 CEO，玩法完全不同。

### 5.3 满意度日常波动

| 事件 | 影响 |
|------|------|
| 股价单季度涨 >10% | +3 |
| 股价单季度跌 >10% | -5 |
| 公司连续亏损 2 季度 | -8 |
| 放弃决策（跳过季度） | -2 |
| 公司被 SEC 罚款 | -10 |
| 行业景气度 < 0.90 | -3 (行业低迷，不怪你但也不高兴) |
| 超额完成年度 KPI | 年度考核时 +10 |

## 七、行业差异化模型

### 7.1 天花板与驱动力公式

```
实际产出 = MIN(驱动力数量 × 单位产出, 天花板数量 × 单位上限)
```

| 行业 | 天花板 | 单位产出 (每人/季) | 单位上限 (每组/季) | 建造成本 | 建造周期 | 特殊规则 |
|------|--------|-------------------|-------------------|---------|---------|---------|
| 制造 | 产线 | 2,000 件 | 10,000 件 | ¥80,000 | 1 季 | — |
| 矿业 | 探明储量 | 1,500 单位 | 储量 × 20%（上限） | ¥120,000 | 2 季 | 开采消耗储量，越挖越少 |
| 科技 | 服务器 | 500 用户 | 5,000 用户 | ¥50,000 | 0 季 | 5 季贬值 30% |
| 金融 | 牌照 | ¥200 万 | ¥1,000 万 | ¥200,000 | 3 季 | — |
| 消费 | 门店 | 150 人次 | 800 人次 | ¥60,000 | 1 季 | — |
| 医疗 | 实验室 | 0.5 个项目 | 3 个项目 | ¥150,000 | 2 季 | 产出随机 |

### 7.2 制造业供需模型

制造业和矿业采用**实物供需模型**——生产量、销售量、库存、营收由工人和天花板通过完整链条推导得出。其他行业（科技/金融/消费/医疗）营收直接 = 员工 × 人均产出 × 景气度系数，实物供需留待后续迭代。

```
━━━━━━ 制造业完整公式链 ━━━━━━━

销售单价  = ¥20 × 景气度^0.6                         ← 景气期卖得贵，衰退期贱卖

生产量    = MIN(工人 × 2,000 件, 产线条数 × 10,000 件)   ← 工人和产线双约束

销售量    = MIN(生产量 + 上季库存, 季度需求)              ← 库存优先出清

营收      = 销售量 × 销售单价                             ← 卖多少赚多少

库存      = MAX(0, 上季库存 + 生产量 - 销售量)             ← 卖不掉的堆起来

仓储费    = 库存 × ¥0.5/件                                ← 堆货有成本

基础维护  = 全部产线 × 基础维护费率                       ← 含闲置产线，每条都要保养

运营成本  = 开工产线 × 运营成本费率                       ← 仅开资产线的额外消耗

人工成本  = 工人数 × ¥2,500/季                            ← 工资

总成本    = 人工成本 + 基础维护 + 运营成本 + 仓储费

净利润    = 营收 - 总成本
```

**成本费率**：每条产线有两个费率，对应拆分开销：
| 费项 | 计费对象 | 制造例子 | 含义 |
|------|---------|---------|------|
| `BaseMaintenanceRate` | 全部产线 | ¥1,000/季 | 折旧、防锈、基础保养 |
| `OperationalCostRate` | 仅开工产线 | ¥2,000/季 | 进料、电费、运转磨损 |

两者相加等价于旧单字段模型（总维护费 = active×3,000 + idle×1,000），拆分后语义更清晰。

**核心博弈**：配平产线 + 工人 + 需求三者。产能过剩 → 库存堆积 → 仓储费吃利润；产能不足 → 丢需求 → 营收触顶。

**起始状态**：
```
起始工人 = 5 人
起始产线 = 1 条（上限 10,000 件/季）
起始需求 = 5 × 2,000 × random(50%, 80%)
          = 5,000 ~ 8,000 件/季           ← 每人不同，留 20-50% 扩张空间
```

#### 矿业开采模型

```
━━━━━━ 矿业完整公式链 ━━━━━━━

销售单价  = ¥12 × 景气度^1.2                              ← 价格弹性最高，景气波动被放大

季度上限  = ⌈探明储量 × 20%⌉                               ← 每季最多开采储量的 20%（向上取整）

生产量    = MIN(工人 × 1,500 单位, 季度上限)               ← 工人和储量双约束

销售量    = MIN(生产量 + 上季库存, 季度需求)                ← 库存优先出清

营收      = 销售量 × 销售单价                              ← 卖多少赚多少

库存      = MAX(0, 上季库存 + 生产量 - 销售量)              ← 可屯矿等高价

仓储费    = 库存 × ¥0.3/单位                               ← 矿石储存成本低

基础维护  = 储量 / 100 × ¥1                                ← 矿坑排水通风等基本开销

运营成本  = 开工工人 × ¥1,200                              ← 爆破、运输等开采消耗

人工成本  = 工人数 × ¥2,500/季                             ← 工资

剩余储量  = 上季储量 - 生产量                              ← 挖一点少一点

总成本    = 人工成本 + 基础维护 + 运营成本 + 仓储费

净利润    = 营收 - 总成本
```

**成本费率**：

| 费项 | 计费对象 | 矿业数值 | 含义 |
|------|---------|---------|------|
| `BaseMaintenanceRate` | 探明储量 ÷ 100 | ¥1/百单位 | 矿坑通风、排水、安全维护 |
| `OperationalCostRate` | 开工工人 | ¥1,200/人 | 爆破、运输、机械运转 |

**核心博弈**：储量递减带来自然产量衰减——越挖越少、越挖越慢。玩家需持续投入勘探（¥120,000/次，2季见效）补充储量以维持产量。勘探结果随机（70%贫矿 2-6万 / 20%中矿 6.5-11万 / 10%富矿 11-18万），期望 6 万单位。采掘资产按 ¥2.0/单位估值（¥120,000 ÷ 60,000 单位），勘探期望投入等于期望产出，资产价值守恒。屯矿待景气高位抛售是核心策略。

**起始状态**：
```
起始探明储量 = 50,000 单位
起始工人 = 5 人
起始季度上限 = 50,000 × 20% = 10,000 单位/季
起始需求 = 5 × 1,500 × random(50%, 80%)
         = 3,750 ~ 6,000 单位/季
```

**产量衰减示例**（5工人，无新勘探）：

| 季度 | 储量 | 20%上限 | 实际产量 |
|------|------|---------|---------|
| 0 | 50,000 | 10,000 | 7,500 |
| 1 | 42,500 | 8,500 | 7,500 |
| 2 | 35,000 | 7,000 | 7,000 |
| 3 | 28,000 | 5,600 | 5,600 |
| 4 | 22,400 | 4,480 | 4,480 |

第3季度起20%上限绑定，产量逐季衰减，工人不再满负荷。

### 7.3 需求侧：行业景气度

景气度用**连续游走**替代"繁荣/正常/衰退"三档。每个季度每个行业都有一个景气度值（0.70~1.30），它决定了：

- **行业需求增长率**：公司当季需求 = 上季需求 × (景气度 + 个体波动)
- **销售单价**：单价 = 基础单价 × 景气度^价格弹性

#### 景气度游走公式

每一季景气度做一次「随机漫步 + 回归拉力」：

```
偏离率   = (当前景气度 - 1.0) / 半区间           // -1.0 ~ +1.0，偏离中心多远
回归占比 = |偏离率| × 回归强度                   // 越极端，回归越强
随机占比 = 1 - 回归占比                           // 剩余交给随机

随机步   = random(-maxStep, +maxStep) × 随机占比
回归步   = -偏离率 × maxStep × 回归占比           // 负号 = 向 1.0 拉
本季变化 = clamp(随机步 + 回归步, -maxStep, +maxStep)

新景气度 = clamp(旧景气度 + 本季变化, 下限, 上限)
```

效果：景气度在上下限间缓慢起伏，极端值自动回归，不会暴涨暴跌。

#### 各行业景气度参数

| 行业 | 范围 | 最大步长 | 回归强度 | 价格弹性 | 个体波动 | 一句话 |
|------|------|---------|---------|---------|---------|--------|
| 制造 | 0.80~1.20 | 0.05 | 0.15 | 0.6 | ±10% | 温和工业周期 |
| 矿业 | 0.72~1.28 | 0.07 | 0.10 | 1.2 | ±10% | 大宗商品过山车 |
| 科技 | 0.65~1.35 | 0.09 | 0.06 | 0.2 | ±10% | 风口与寒冬交替 |
| 金融 | 0.75~1.25 | 0.06 | 0.12 | 0.4 | ±10% | 杠杆放大周期 |
| 消费 | 0.88~1.12 | 0.04 | 0.20 | 0.5 | ±10% | 稳如磐石 |
| 医疗 | 0.88~1.16 | 0.04 | 0.25 | 0.3 | ±10% | 衰退免疫 |

**景气度存储**：`IndustryProsperity` 表，`(industry, quarter, prosperity)` 联合唯一，写入每季游走结果。公司需求计算查最新一行。

**每家公司内再分化**：

```
公司需求增长率 = 行业景气度 + 个体波动(±10%)

  景气度=1.15 (+15%), 个体+8% → 需求×1.23  (旺)
  景气度=0.88 (-12%), 个体+9% → 需求×0.97  (个体扛住)
  景气度=1.15 (+15%), 个体-9% → 需求×1.06  (景气旺，个体不行)
```

**营销**：✅ 已实现。投入 ¥N（滑块自选）→ 当季需求 +round(N × random(Min, Max))。制造 Min/Max = 0.075/0.175（¥20,000 → 1,500~3,500），矿业 = 0.125/0.292（¥20,000 → 2,500~5,833）。效果在结算前直接写入 `Company.Demand`，结算时受 `2×(产能+库存)` 封顶截断。一次性生效，不跨季。

**产能封顶**：需求增长存在上限，不得超过当前可交付总量（产能 + 库存）的 2 倍：

新需求 = MIN(上季需求 × (景气度 + 个体波动), (季度产能 + 上季库存) × 2.0)
```
（营销效果在结算前直接写入 `Company.Demand`，与自然增长合并后统一受此封顶，不再单独计算）

- 库存充足的公司在产能受限时仍能靠库存供货，需求不因此萎缩
- 库存耗尽后封顶自动收紧至纯产能 × 2.0，防止持续透支
- 衰退期（景气度 < 1.0）不受封顶影响，需求正常萎缩

### 7.4 天花板维护费

所有行业的天花板单元每季产生维护费。使用中 > 闲置（使用中有运营损耗，闲置仅基础保养）。

| 行业 | 天花板 | 使用中/条/季 | 闲置/条/季 | 新建/条 | 维护/新建比 |
|------|--------|------------|-----------|---------|------------|
| 制造 | 产线 | ¥3,000 | ¥1,000 | ¥80,000 | 2.5% ~ 3.8% |
| 矿业 | 探明储量 | ¥1/百单位 | —（储量耗竭无需维护） | ¥120,000 | — |
| 科技 | 服务器 | ¥2,500 | ¥800 | ¥50,000 | 3.2% ~ 5.0% |
| 金融 | 牌照 | ¥6,000 | ¥3,000 | ¥200,000 | 2.5% ~ 3.0% |
| 消费 | 门店 | ¥3,000 | ¥1,500 | ¥60,000 | 4.2% ~ 5.0% |
| 医疗 | 实验室 | ¥5,000 | ¥2,000 | ¥150,000 | 2.7% ~ 3.3% |

**闲置比使用中贵的反常识已纠正**：使用 = 正常损耗 + 运营消耗（电费/原料/合规）；闲置 = 基础保管（防锈/巡检/年费）。消费门店闲置最重（空店极丑），科技服务器闲置比最高（空转的服务器白养）。

### 7.5 各行业风格差异

| 行业 | 一句话 | 天花板特点 |
|------|--------|-----------|
| 制造 | 预测需求，建产线，别屯货 | 永久，需维护费，库存是双刃剑 |
| 矿业 | 储量越挖越少，勘探续命，屯矿等高景气 | 每季最多开采20%，储量递减，越挖越慢 |
| 科技 | 烧钱抢用户，旧设备贬值 | 便宜但 5 季贬值 30%，景气最狂野 |
| 金融 | 看准市况加杠杆，亏了自己扛 | 最贵最慢，景气敏感 |
| 消费 | 持续营销保持热度 | 需每季维护，品牌冷即客流断 |
| 医疗 | 前几年不赚钱，后面专利收不停 | 慢，产出随机，景气最抗跌 |

## 八、研发系统

### 8.1 通用机制

```
研发 [1AP, ¥20,000] → 技术等级 +1 → 永久生效

成本递增：下次 ¥30,000 → ¥45,000 → ¥70,000 → ¥100,000...
无上限，无进度条，投入即生效
科技/医疗 研发享受 1.5 倍效果
```

### 8.2 各行业研发产出（间接收益模式）

每级降低该行业的运营惩罚，让错误的代价越来越小：

| 行业 | 研发产出 | 每级效果 |
|------|---------|---------|
| 制造 | 精益生产 | 库存仓储费 -20%（5 级后库存免费存） |
| 矿业 | 高效开采 | 季度开采上限 +2%（即储量 × 22%，逐级递增） |
| 科技 | 自动运维 | 闲置服务器维护费 -25%/级 |
| 金融 | 风控模型 | 坏账率 -0.5%/级（市况差时少亏） |
| 消费 | 品牌资产 | 品牌冷却速度 -20%/级 |
| 医疗 | 先导化合物 | 新药失败成本退回 30%/级 |

**设计原则**：研发不直接提升利润率。它让玩家犯错的代价变低——建多了产线？仓储费减了。服务器闲置？维护费降了。品牌忘记营销？冷却慢了。越研究越从容。

## 九、季度结算流程

```
季度开始 (cron tick, 每 5 分钟整点)
  │
  ├─ 1. 景气度更新 (瞬间完成)
  │     每行业游走计算 → 写入 IndustryProsperity 表
  │     写入后所有公司当季景气度已确定
  │
  ├─ 2. 批量基准结算 (后台 goroutine，分片处理)
  │     遍历所有活跃公司：
  │       - 如本季快照已存在（被决策路径抢先）→ 跳过
  │       - 否则 → SettleBaseline() → INSERT CompanyQuarterly
  │     纯函数结算，无副作用。分片退让，不阻塞 API。
  │
  ├─ 3. 决策阶段 (5 分钟窗口)
  │     CEO 打开面板 → 读取已生成的 baseline 快照
  │     CEO 提交决策 → 带决策重算 → UPDATE 本季快照
  │     可跳过 (董事会 -2 满意度)
  │
  └─ 4. 季度公告 (Q 结束时)
        向所有玩家广播公司财报
        如果是年度末 (Q4)，考核董事会 KPI

两条路径生成快照（事务行锁保证幂等）：
  - 后台批量：INSERT baseline
  - CEO 决策：先查是否存在 → 不存在则先算 baseline → 再带决策重算
  谁先到谁创建，后到的跳过创建步骤。
```

## 十一、股价公式与总资产

> **前置概念**：P3 交易引擎上线前，发行价和理论股价均使用以下公式。P3 正式运营后，市场价格由撮合引擎产出，本公式仅作为理论锚定价（防止市场价与基本面脱锚）和 IPO 发行价计算依据。

### 11.1 总资产

```
总资产 = Cash + CapCount × CapAssetValue
```

各行业 `CapAssetValue` 映射此行业天花板的会计估值：

| 行业 | CapCount 含义 | CapBuildCost | CapAssetValue | 计算依据 |
|------|-------------|-------------|---------------|---------|
| 制造 | 产线数 | ¥80,000 | ¥80,000 | = CapBuildCost |
| 矿业 | 探明储量（吨） | ¥120,000/次 | ¥2.0 | = 120,000 ÷ 60,000（单次勘探期望） |
| 科技 | 服务器数 | ¥50,000 | ¥50,000 | = CapBuildCost |
| 金融 | 牌照数 | ¥200,000 | ¥200,000 | = CapBuildCost |
| 消费 | 门店数 | ¥60,000 | ¥60,000 | = CapBuildCost |
| 医疗 | 实验室数 | ¥150,000 | ¥150,000 | = CapBuildCost |

> **矿业推导**：勘探一次期望探明 60,000 单位（70%×40k + 20%×87.5k + 10%×145k = 60k），成本 ¥120,000，因此每单位估值 ¥2.0。勘探投入等于期望产出的资产价值，达到「资产保值」——贫矿微跌、中矿持平、富矿微涨。

### 11.2 股价公式

```
NAV       = 总资产 / TotalShares
EPS       = 近4季度平均净利润 / TotalShares
理论股价   = max(1, NAV + EPS × 行业PE × 景气度)
发行价     = 理论股价 × 0.95                     （IPO 承销折扣）
```

> **为什么是 NAV + EPS×PE**：NAV 代表公司清算价值（地板价），EPS×PE 代表未来盈利能力的估值溢价。持续盈利的公司 PE 溢价高，盈利差的公司股价向净资产回归。

PE 并非固定——基准 PE 仅作起点，公司经营表现影响估值：
- 营收增长率 > 20% → PE 临时 +20%
- 营收增长率 < -10% → PE 临时 -15%
- 库存/营收 > 50% → PE 临时 -15%（制造业滞销折价）
- 连续 2 季亏损 → PE 临时 -20%

### 11.3 行业基础参数

| 行业 | 基准PE | 人均营收/季 | 起始员工 | 起始现金 | CapBuildCost | CapAssetValue |
|------|-----|-----------|---------|---------|-------------|---------------|
| 科技 | 25 | ¥40,000 | 3 | ¥20,000 | ¥50,000 | ¥50,000 |
| 金融 | 12 | ¥50,000 | 3 | ¥50,000 | ¥200,000 | ¥200,000 |
| 制造 | 12 | ¥40,000 | 5 | ¥30,000 | ¥80,000 | ¥80,000 |
| 矿业 | 10 | ¥35,000 | 5 | ¥50,000 | ¥120,000 | ¥2.0 |
| 消费 | 20 | ¥35,000 | 3 | ¥15,000 | ¥60,000 | ¥60,000 |
| 医疗 | 30 | ¥50,000 | 3 | ¥40,000 | ¥150,000 | ¥150,000 |

**设计原则**：
- 制造/矿业 人多、股本大、PE 低 → 劳动/资源密集型，股价靠量
- 科技/医疗 人少、股本小、PE 高 → 智力密集型，股价靠质
- 金融/消费 介于中间

### 14.1 数据模型

```
Company 表核心字段：
  CEOID, Symbol, Name, Industry, Cash, Employees, CreatedQuarter, LastSettledQuarter, Status
  CEOShares      — CEO 持股（固定 100,000 股）

  操作记录（Actions, JSON）
```

### 14.2 融资创建流程（v3 — 2026-06-23 修订）

```
玩家初始现金: 100,000

创建参数:
  investor_shares   — 投资方持股数（10万-190万，滑动条）
  player_investment — 玩家出资额（1,000-可用现金，滑动条）

固定值:
  ceo_shares = 100,000 股

派生值:
  total_shares  = 100,000 + investor_shares
  own_ratio     = 100,000 / total_shares          (5%-100%)
  company_cash  = player_investment / own_ratio   (含社会/投资方融资)
  investor_cash = company_cash - player_investment

行业控制:
  IndustryConfig.Enabled — 行业开放开关，逐步开放各行业
  当前创建 API 对未开放行业返回"该行业暂未开放"

初始资源:
  制造业: CapCount = 1 (免费产线), Employees = StartingEmployees
  矿业:   通过首次预结算设置 CapCount = MiningInitialReserves(50,000)

IPO:
   公司创建后不可在二级市场交易（IpoQuarter = 0），需达成 IPO 条件后方可上市。
   IPO 时 CEO 持股转为个人持仓、投资方持股与增发股一同进入证券机构库存。
   详见 §十六。
```

## 十六、IPO 与二级市场交易 (P3)

> **位置**：公司运营（P2）完整后，P3 交易引擎上线。本节描述 IPO 机制与二级市场衔接的完整设计。

### 16.1 时间体系

| 参数 | 值 |
|------|-----|
| 交易 tick | 2 秒 |
| tick/季 | 150 tick |
| 1 季度 | 5 分钟 |
| K 线周期 | `"40t"`（~1.3min）、`"150t"`（1季）、`"600t"`（1年） |

### 16.2 股份结构模型

创建公司时玩家直接输入投资方持股（不再是总股本），总股本为派生值：

```
创建时:
  CEOShares      = 100,000  (固定)
  InvestorShares = X        (玩家输入，10万-190万股)
  TotalShares    = 100,000 + X

IPO 增发后:
  PublicFloat    = TotalShares(IPO前) × 增发比例
  TotalShares    = 100,000 + InvestorShares + PublicFloat
```

三种股份的 IPO 后流向：

| 类型 | 来源 | IPO 后流向 | 交易规则 |
|------|------|-----------|---------|
| CEO 持股 | 创始人，固定 100,000 股 | 转为 CEO 玩家的个人股票持仓（Holdings 表），AvgCost = IPO 发行价 | 二级市场自由交易，与普通持仓无异 |
| 投资方持股 | 公司创建时的外部融资方 | 与 IPO 增发股一起进入证券机构库存（BrokerInventory） | 由证券机构逐步释放到市场 |
| 流通股 | IPO 增发 | 进入证券机构库存（BrokerInventory） | 由证券机构逐步释放到二级市场 |

> **设计简化**：无锁定期、无减持/增持 action。上市即解禁，所有股份纳入同一交易体系。投资者可通过市场页看到各股票在 BrokerInventory 中的待释放库存量。

### 16.3 IPO 条件

| 条件 | 值 |
|------|-----|
| 运营季度 | ≥ 12 季度（3年） |
| 连续盈利 | ≥ 4 季度 |
| 现金 | ≥ ¥1,000,000 |
| 年度营收 | 近 4 季度合计 ≥ ¥5,000,000 |

### 16.4 IPO 执行（即时）

```
CEO 发起 IPO action → 选择增发比例（滑动条，10%-50%）

增发股数   = TotalShares(IPO前) × 增发比例
发行价     = round((NAV + EPS × 行业PE × 景气度) × 0.95 × 100)  分

公司获得现金 = 增发股数 × 发行价(元) → Company.Cash += 募资额
流通股      = 增发股数 → 全部进入 BrokerInventory
IpoQuarter  = 当前全局季度号
```

IPO 即时执行，不等待季度结算。

### 16.5 证券机构库存释放机制

IPO 增发的全部流通股先进入证券机构（BrokerInventory 表），由其逐步释放到二级市场：

```
每 N ticks (建议5) 扫描一次:
  ├── 遍历所有已上市 Stock
  ├── 找出存在超过 M ticks (建议10) 仍未成交的限价买单
  └── 若 BrokerInventory.可用 > 0:
       ├── 按买一价(优先)或当前卖一价成交
       ├── BrokerInventory.TotalQty -= 成交量
       └── 买方 → 持仓增加
```

`BrokerInventory.FrozenQty` 字段预留未来做空融券使用，当前固定为 0。

### 16.6 交易模型：限价单与市价单

| | 限价单 (Limit) | 市价单 (Market) |
|---|---|---|
| 下单时 | 指定价格和数量 | 只指定数量 |
| 立即成交？ | 不一定，挂入订单簿等待 | 立即按最优价位扫单 |
| 价格保护 | 有，不会超预算 | 无，可能滑点 |
| 典型场景 | "¥12.50 我才买" | "立刻成交，不管多少" |

撮合规则：价格优先 → 同价时间优先（SeqNum）。市价单按当前最优价吃完为止。

### 16.8 数据库新增表

| 表 | 用途 |
|------|------|
| `Stock` | 上市股票信息：实时价格/涨跌/PE/NAV/成交量/盘口五档（Company 1:1） |
| `Order` | 订单簿：限价单/市价单，状态机 open→partial→filled/cancelled |
| `Trade` | 成交记录（重写原 Transaction 表）：买卖双方 + 订单引用 |
| `Candle` | K 线聚合：(StockID, Period, OpenTime) 唯一键 |
| `BrokerInventory` | 证券机构库存：TotalQty（可用库存）/FrozenQty（预留融券） |

现有 `Holding` 表改为 `StockID` 外键替代字符串 `Symbol`；现有 `Transaction` 重写为 `Trade`。

### 16.9 Symbol 生成规则

```
前缀 + 行业自增序号
行业前缀: tech=TK, finance=FI, manufacturing=MF, mining=MN, consumer=CS, healthcare=YL
示例: MF001, MF002, MN001, TK001...
```

按行业独立计数，查询 `SELECT MAX(symbol) FROM companies WHERE symbol LIKE 'MF%'` 取最大后 +1。数据库 uniqueIndex 兜底并发冲突。
